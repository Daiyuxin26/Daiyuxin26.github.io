01
Project One / CRC Copilot

把散落的产研管理，做成一套可追踪、可约束、可复盘的系统。

CRC Copilot 的核心不是做一个“需求列表”，而是把产品、研发、测试、管理者之间的协作过程抽象为状态机、权限模型、看板与 Agent 分工。它回答两个问题：现在是什么情况，以及为什么会这样、接下来怎么办。

  • 01.1

    先把产研协作的混乱边界拆清楚。

    原流程里，需求文档、开发任务、测试记录、延期原因和管理看板分散在不同系统，管理层很难实时判断项目健康度，一线同学也很难知道下一步该由谁推进。

    Pain Point
    信息分散

    需求、任务、日志、问答和周报不在同一套协作语境中。

    Pain Point
    状态不清

    需求从规划到上线缺少强约束流转，延期和评审原因不稳定沉淀。

    Product Goal
    全流程可追溯

    用系统规则替代口头约定，让过程数据可查、可看、可复盘。

  • 01.2

    产品架构：状态流 + RBAC + 看板 + Agent。

    系统围绕 PM、RD、QA、Admin 四类角色设计权限边界；把需求生命周期标准化为从初始化、方案中、待评审、已排评审、评审通过、已排期、研发中、测试中、验收完成到上线的状态路径。

    需求创建 → 评审排期 → 研发拆分 → 测试验收 → 上线复盘
    Board

    看板回答“现在是什么情况”

    • 需求状态、延期、阻塞、负责人、里程碑一屏可见。
    • 管理者可以快速定位高风险需求与高负载角色。
    Agent

    Agent 回答“为什么、怎么办”

    • 用自然语言追问延期原因、瓶颈阶段、人员负载和优先级。
    • 输出调度建议、风险判断和周度总结。
  • 01.3

    Vibe Coding 0-1：从 PRD 到前后端、看板和部署验证。

    在研发资源受限的情况下，我用 Vibe Coding 独立完成 MVP 和多个核心模块落地，覆盖需求定义、信息架构、交互原型、Vue 前端、Go/Gin 后端、数据库字段、看板、部署与 SOP。

    Core
    需求管理

    创建、编辑、状态流转、评审、延期原因、操作日志。

    Collab
    待办 / 问答

    把任务分派、问题指定、答案沉淀做成结构化协作资产。

    Insight
    数据看板

    需求趋势、延期分布、人员负载、里程碑进度可视化。

    Automation
    智能周报

    把部门周报、反馈、总结和问题追踪接入统一工作台。

  • 01.4

    结果：系统上线后开始承接真实产研管理。

    5 项目 / 453 需求 / 17.5% 延期率下降
    这个项目证明了：PM 不只是写需求，也可以用 AI 把需求系统本身做出来。
    Product manager as system builder
    • CRC Copilot 产研管理系统：在研发资源受限背景下，基于大模型与 Vibe Coding 独立完成 AI 产研管理系统 MVP，覆盖需求定义、信息架构、交互原型、前后端模块、看板搭建与部署验证。
    • 产研流程抽象：梳理 PM / RD / QA / 管理层多角色工作流，将需求规划、评审、拆分、开发、测试、上线抽象为结构化状态流，并设计延期原因、评审结果、操作日志等强留痕机制。
    • 业务效果：系统上线后覆盖 5 个项目、453 个需求，支撑 15 个里程碑与 316 条日常迭代需求管理，需求延期率下降 17.5%。

02
Project Two / Business Management Agent

把组织协作从“人找信息”，改成 Agent 主动拉齐上下文。

业务管理 Agent 是企业内部 AI Agent 组织协作平台。它不只是聊天机器人，而是由 LangGraph 智能体后端、Web 聊天前端、FastAPI 业务工具层、钉钉机器人、沙盒与 Skill 管理后台、Supervisor 巡检服务和运维控制台组成的多服务系统。

  • 02.1

    问题不是缺一个入口，而是组织上下文太分散。

    项目管理里大量时间耗在跨系统查询、反复同步、周报整理、任务追踪和风险提醒。这个项目把“人-人同步协作”升级为“人-Agent-人”：Agent 负责搜集、编排和提醒，人负责判断和确认。

    Users
    部门全员

    覆盖产研测、算法、设计师、业务需求方和管理层。

    Entry
    Web + 钉钉

    同一套 Agent 能力接入 Web 聊天与企业 IM，降低使用门槛。

    Mode
    人-Agent-人

    信息整合、任务编排、进度追踪和风险提醒由 Agent 承接。

  • 02.2

    总体架构：多入口统一到 LangGraph Agent。

    Web 用户和钉钉用户进入同一套 LangGraph / deepagents 后端；Agent 通过 agent-hand 调用业务 API，通过沙盒维护用户记忆和 Skill，通过 Supervisor 做后台巡检，通过 ops-server 做多服务运维。

    Client
    Web / DingTalk
    自然语言查询、任务协作、报告生成、IM 推送。
    Agent Core
    LangGraph
    Thread、Cron、Memory、Tool Calling、动态 Skill。
    Business Hand
    FastAPI
    项目、需求、日志、待办、任务、RMS 等业务工具层。
    Ops Layer
    Admin / Supervisor
    Skill 权限、记忆管理、巡检、通知、部署、日志。
  • 02.3

    产品亮点：自然语言变成组织协作动作。

    用户可以用自然语言查询项目、需求、人员、待办、日志，生成项目分析、周报、总结和可视化结果；也可以创建父子任务、派发给多人、进入任务专属会话并沉淀结果。

    Information

    查询、解释、生成报告

    • SQL 查询只允许 SELECT，减少 Agent 直接写库风险。
    • 支持项目分析、人员工作分析、周报、图表和总结。
    • 长期记忆与历史事件分层存储，避免上下文爆炸。
    Action

    创建、派发、追踪任务

    • 自然语言创建父任务和多个子任务。
    • 任务绑定 LangGraph thread，执行人进入专属上下文。
    • 状态、结果、对话、通知和审计形成闭环。
  • 02.4

    工程亮点：动态 Skill、沙盒记忆和可运营后台。

    项目扩展了 deepagents 的 Skill 注入机制：每次 LLM 调用前扫描系统 Skill 和用户 Skill，根据角色权限动态过滤；沙盒保存用户记忆和工作区；管理后台支持 Skill 热管理、记忆编辑、运行监控和权限配置。

    Skill
    动态注入

    系统 / 用户 Skill 分层，角色级权限过滤，新增或编辑后即时生效。

    Memory
    双层记忆

    MEMORY 注入上下文，HISTORY 只追加并通过检索查询。

    Reliability
    企业化封装

    沙盒 TTL/LRU、Graph cache 失效、Cron 校验、通知去重、日志轮转。

  • 02.5

    结果：管理效率提升，查询和周报降到分钟级。

    60% 管理效率提升 → 分钟级 周报 / 查询
    这个项目的价值不是“问答更方便”，而是让 Agent 成为组织协作的中间层。
    Agent as coordination layer
    • 业务管理 Agent：主导企业内部 AI Agent 组织协作平台 0-1 设计与迭代，探索“人-Agent-人”智能协作模式，由 Agent 承接信息整合、任务编排、进度追踪和风险提醒等重复工作。
    • 多入口 AI 助手：设计 Web + 钉钉多入口 AI 工作助手，支持自然语言查询项目、需求、人员、待办、日志，并自动生成项目分析、周报、总结和可视化结果。
    • Agent 平台化能力：围绕 LangGraph / deepagents 构建多 Agent 后端、FastAPI 工具服务、沙盒化 Skill、动态权限、长期记忆、定时任务、Supervisor 巡检和运维控制台。
    • 上线效果：项目覆盖部门全员及跨部门协作方，管理层反馈整体管理效率提升约 60%，周报生成和项目数据查询耗时降低至分钟级。

03
Main Project / PM Agent Runtime

把 AI 协作从聊天记录，改造成可恢复、可审计的产品工作流。

PM Agent 是面向 Claude Code / Codex / Kiro 的 AI Agent 产品管理工作流系统。它不是 PRD 模板库，而是把需求澄清、调研、PRD、外部信号观察、多 Agent 辩论和交付导出，统一落到本地文件协议、状态机和 CLI 前门里。

  • 03.1

    项目定位：Agent 操作系统里的产品管理层

    长期 AI 协作的难点不是“模型会不会写”，而是它不知道项目当前状态、哪些结论已经稳定、哪些只是临时讨论。PM Agent 用 `status / route / review / start / next / resume` 作为稳定前门，让外部 Agent 每次进入项目时先恢复状态，再继续推进。

    Role
    0-1 Owner

    主导产品架构、工作流设计、信息架构和 Vibe Coding 实现。

    Agent Surface
    Claude / Codex / Kiro

    围绕多种外部 Agent CLI 设计统一执行器和工作流入口。

    Quality
    174 passed

    pytest 覆盖 CLI、Observation、Debate、Hooks、Scheduler、Readiness 等模块。

  • 03.2

    产品架构：Project / Workspace 双层模型

    Project 存长期战略、决策、记忆和 observation policy；Workspace 存单轮需求、Requirement、Research、PRD、candidate updates、maintenance 和 debate artifacts。这样长期知识和本轮执行状态不会混在一起。

    Project Layer

    长期知识

    • 长期战略、背景资料、决策记录
    • 项目记忆与跨 workspace 可复用信息
    • Observation policy / state 的项目级归属
    Workspace Layer

    单轮交付

    • Requirement、Research、PRD 与 testcase
    • 候选 observation updates 与 maintenance draft
    • Debate round artifacts、synthesis 与 review 状态
  • 03.3

    工作流入口：让 Agent 不再猜下一步

    01
    status

    读取当前 workspace 状态、phase、pending decision 与 backlog。

    02
    route

    判断当前应该进入澄清、调研、PRD、维护或 review 面。

    03
    review

    打开当前最重要的候选评审、辩论评审或维护视图。

    04
    next

    输出下一步建议，降低多轮协作里的方向漂移。

    05
    resume

    跨会话恢复项目，明确继续点和可写边界。

  • 03.4

    Debate 机制：让重大产品判断先暴露张力

    Debate 是 PM Agent 的决策增强旁路。主 Agent 发起辩论，但不亲自下场；pmagent orchestrator 负责编排 defender / attacker / synthesizer、维护 session、落盘 round artifacts，并把 synthesis 通过 review / resolve 回流主流程。

    Debate Flow

    3 轮结构化对打

    • Round 0：独立起手，避免被对方叙事框住
    • Round 1：交锋反驳，攻击对方最弱假设
    • Round 2：聚焦最无法调和的分歧点
    • Synthesis：汇总收敛点、核心分歧、意外发现与下一步
    Governance

    Side-channel，不直接改 PRD

    辩论产物落在 `context/debates/`，属于 context artifact；它不直接修改 Requirement / PRD / decisions，而是先进入 review surface，由人类接受、拒绝或延后。

    Debate 的价值不是“谁赢了”，而是让人类裁判看见哪些判断还没打穿。
    Convergence / Divergence / Unexpected Findings
  • 03.5

    Observation 机制：把静态项目变成动态维护的项目知识库

    项目启动后，市场、竞品、用户反馈和技术生态会持续变化。Observation 的重点不是单次搜索，而是让 project-level knowledge 持续吸收外部信号，并通过评审和维护机制，把“项目资料夹”变成会更新、可追溯、可被 Agent 复用的动态知识库。

    Static
    项目资料

    Requirement、PRD、决策、调研和背景信息原本只是静态文件。

    Signal
    外部变化

    Observation 持续把新市场信号转成可追踪 observation records。

    Curation
    评审筛选

    候选信号先被接受、拒绝或延后，避免噪声直接污染主线。

    Living KB
    动态知识库

    稳定结论回流项目记忆和维护草稿，让后续 Agent 接手时读到最新上下文。

    Observation 的核心，是让项目知识库跟着外部世界一起变新。
    Static project files → Living project knowledge base
    • AI Agent 产品管理工作流系统：主导设计并落地面向 Claude Code / Codex / Kiro 的 PM Agent Runtime，覆盖需求澄清、调研、PRD、外部信号观察、候选变更评审、多 Agent 辩论和交付导出。
    • 产品架构与工作流抽象：设计 project / workspace 双层信息架构、phase 状态机、readiness gate，以及 status / route / review / start / next / resume 前门命令，将 AI Agent 产品协作过程转化为可恢复、可审计、可持续迭代的本地文件协议。
    • Debate 决策机制：构建 defender / attacker / synthesizer 多 Agent 辩论闭环，通过 3 轮结构化对打、round artifacts、synthesis、review / resolve 和失败状态回流，让重大产品判断先暴露分歧与盲区。
    • Observation 观察机制：构建项目知识维护闭环，通过 Agent-delegated 搜索、raw findings ingest、project-level observation log、workspace candidate review、accept / reject / snooze 和 maintenance draft，把静态项目资料升级为动态维护的知识库。

Project Four / Ops Directory Strategy Agent

把目录运营从人工巡检，改造成 Agent 生成策略、运营确认、系统复盘。

运营目录策略 Agent 面向移动端创意资源目录。它把“查数、诊断、选资源、调目录、看结果”的人工流程抽象为 Agent 可执行任务链，同时保留运营确认权，避免黑盒自动化直接改线上内容。

  • 04.1

    目录是最强曝光杠杆，但人工覆盖已经到顶。

    FR/Music 数据显示，入目录资源只占 9.1%，却贡献 68.3% 导出，平均产出约为未入目录资源的 21.5 倍。问题是目录规模达到 7 品类 × 320 目录 × 9 语言，单语言全量处理就需要约 266 小时。

    24,191 资源 → 21.5x 目录生产力倍差
    Waste
    603 条占坑

    目录内 E 级零导出资源占坑不产出，浪费曝光位。

    Pool
    22,298 条 E+N

    沉底资源池规模大，其中部分低曝光资源已经证明有导出能力。

    Constraint
    266 小时

    人工无法覆盖全部目录和语言，必须转为系统化巡盘。

  • 04.2

    Agent 操作模型：换出、换入、调序。

    我将目录优化拆成三个动作：换出高曝光低导出或零导出占坑资源；从 E+N 池中筛选低曝光有潜力资源换入；针对未接入算法排序的目录，识别位置错配并生成调序方案。

    Scan
    自动巡盘
    读取 QT / BI / RMS 数据，生成目录健康度报告卡片。
